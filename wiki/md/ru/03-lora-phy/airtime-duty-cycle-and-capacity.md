# Airtime, duty cycle и ёмкость сети

Airtime — время, в течение которого радиосигнал одного пакета занимает канал. Для mesh это базовая единица стоимости: каждый hop создаёт новую передачу, flood создаёт несколько конкурирующих передач, а ACK проходит обратно.

## Расчёт airtime

Длительность символа:

```text
Tsym = 2^SF / BW
```

Время preamble:

```text
Tpreamble = (Npreamble + 4,25) · Tsym
```

Число payload symbols зависит от:

- длины payload;
- SF;
- CR;
- наличия PHY CRC;
- explicit/implicit header;
- LDRO.

Итог:

```text
ToA = Tpreamble + Npayload_symbols · Tsym
```

Для точного расчёта нужно использовать параметры конкретной сборки. Онлайн-калькуляторы часто предполагают LoRaWAN и добавляют его размер кадра; для MeshCore необходимо вводить фактическую длину сериализованного MeshCore packet.

## Пример для профиля 250 кГц, SF11, CR 4/5

При восьмисимвольной preamble, explicit header и CRC приблизительный airtime:

| LoRa payload | Airtime |
|---:|---:|
| 20 байт | около 330 мс |
| 50 байт | около 575 мс |
| 100 байт | около 944 мс |
| 184 байта | около 1,56 с |

Это только один radio TX. Пакет через три repeater занимает эфир четыре раза в разных точках сети. При flood часть покрытий пересекается, поэтому передачи конкурируют в одном collision domain.

## Полезная длина меньше MeshCore payload limit

MeshCore документирует до 184 байт поля `payload`. Но пользовательский текст должен вместить:

- destination/source hash;
- cipher MAC;
- timestamp;
- flags/attempt;
- padding блочного шифрования;
- сам текст.

В transport route также добавляются 4 байта transport codes, а path растёт с количеством hop. Поэтому «184 байта» не означает 184 символа текста и не гарантирует одинаковый airtime для разных маршрутов.

Unicode требует несколько байт UTF-8. Emoji может занимать четыре и более байт с модификаторами.

## End-to-end стоимость

Для приближённой оценки:

```text
E = ToAdata · Ntx_data + ToAack · Ntx_ack + ToAretry
```

Где `Ntx_data` — число фактических радиопередач исходного пакета. Для direct path из `H` hop без дублей:

```text
Ntx_data = H
```

Если считать hop как линии между узлами. Для flood число зависит от количества повторивших узлов, а не только длины найденного пути.

## Duty cycle

Duty cycle — доля времени передачи в выбранном интервале:

```text
Duty = TX_time / Total_time
```

При 1% устройство может передавать 36 секунд за час. Но юридические правила могут использовать другой интервал, LBT/AFA, dwell time и разные поддиапазоны. Значение проверяется по стране и актуальным документам.

## `dutycycle` в MeshCore

Официальная CLI для firmware v1.15.0+ документирует:

```text
get dutycycle
set dutycycle <1..100>
```

Примеры из документации:

- `100` — без программного лимита;
- `50` — технический default;
- `10` — 10%;
- `1` — 1%.

**Default 50% не является юридической рекомендацией.** Он отражает настройку прошивки. В большинстве нелицензируемых поддиапазонов применимые ограничения могут быть значительно строже.

## Старый airtime factor

Параметр `af` объявлен deprecated начиная с v1.15.0. Его модель:

```text
после TX молчать примерно ToA · af
долгосрочный duty ≈ 1 / (1 + af)
```

Примеры:

| `af` | Приблизительный duty |
|---:|---:|
| 1 | 50% |
| 2 | 33% |
| 3 | 25% |
| 9 | 10% |

Новая настройка `dutycycle` понятнее, но при миграции старых узлов нужно проверить, как прошивка конвертирует сохранённое значение.

## Airtime budget в dispatcher

Текущий dispatcher использует пополняемый budget. После TX фактическое время вычитается. Budget восстанавливается пропорционально прошедшему времени и заданному duty cycle.

Если бюджета мало, следующая передача откладывается. Это влияет на:

- задержку ACK;
- длину outbound queue;
- remote CLI;
- room sync;
- поведение во время flood storm.

Программный budget защищает один узел. Он не координирует суммарный duty всех устройств сети.

## Ёмкость одного collision domain

Теоретически канал свободен 100% времени, но практическая безопасная загрузка намного ниже из-за:

- случайных коллизий;
- hidden nodes;
- preamble detection и turnaround time;
- повторов;
- неодинакового SNR;
- adverts и discovery;
- задержек очереди;
- требований duty cycle.

При высокой загрузке возникает нелинейный эффект: больше пакетов → больше коллизий → больше повторов → ещё больше пакетов. Это congestion collapse.

## Влияние flood

Direct path из четырёх hop создаёт четыре передачи. Flood в плотной области может быть услышан десятью repeaters; случайные задержки и duplicate suppression сократят число повторов, но несколько TX всё равно произойдут.

Параметры, ограничивающие стоимость:

- `flood.max`;
- `flood.max.unscoped`;
- `flood.max.advert`;
- regions/transport codes;
- `txdelay`;
- `rxdelay`;
- loop detection;
- сохранение direct path.

## Интервалы adverts

Advert полезен для обнаружения и обновления path, но является служебным трафиком. Если каждый из 100 узлов отправляет длинный flood advert каждые несколько минут, сеть тратит значительную часть airtime без пользовательских сообщений.

Для стационарного repeater нет необходимости часто подтверждать неизменное существование. Интервал выбирают по:

- мобильности узла;
- скорости устаревания контактов;
- плотности сети;
- длине маршрутов;
- размеру advert;
- region scope.

## Как оценивать сеть

Соберите за репрезентативный период:

- `total_airtime` каждого критического repeater;
- sent/received flood и direct;
- duplicate counters;
- queue length;
- ACK success rate;
- средний размер и hop count;
- интервалы adverts и sensor reports.

Затем разделите трафик на:

1. пользовательский;
2. routing/discovery;
3. подтверждения;
4. повторы;
5. избыточные дубли.

Оптимизация начинается с уменьшения последних двух категорий, а не с ограничения полезных сообщений.

## Снижение airtime

- использовать direct после discovery;
- уменьшать текст и binary payload;
- не отправлять telemetry без изменения значения;
- рандомизировать sensor interval;
- ограничивать flood scope;
- уменьшать advert frequency;
- избегать чрезмерного SF/CR;
- не включать Multi-ACK без причины;
- устранять плохие links, вызывающие retries;
- разделять независимые области по частоте или scope, если это допустимо.
