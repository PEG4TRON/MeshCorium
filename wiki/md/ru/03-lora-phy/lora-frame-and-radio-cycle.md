# LoRa-кадр и цикл работы радиочипа

MeshCore-пакет не передаётся в эфир сам по себе. Прошивка сериализует его в массив байтов и передаёт радиодрайверу. LoRa-трансивер добавляет физическую преамбулу, sync word, PHY header и при необходимости CRC.

![LoRa PHY frame](/attachments/ru/lora-frame.svg?v=2)

## Два вложенных формата

Полезно разделять:

```text
LoRa PHY frame
└── LoRa payload
    └── MeshCore packet
        ├── MeshCore header
        ├── transport codes
        ├── path
        └── MeshCore payload
```

Ошибка LoRa CRC означает, что MeshCore-парсер обычно не получит корректный массив байтов. Ошибка MeshCore MAC возможна уже после успешного LoRa CRC: физический кадр принят, но зашифрованные данные не прошли аутентификацию.

## Preamble

Preamble — последовательность chirp, по которой приёмник обнаруживает LoRa-сигнал и синхронизируется. За ней следуют специальные символы завершения preamble и sync word.

Приёмник может находиться в нескольких состояниях:

1. слушает канал;
2. обнаруживает preamble;
3. подтверждает sync word;
4. демодулирует header;
5. принимает payload;
6. проверяет CRC;
7. формирует IRQ `RX_DONE` или ошибку.

Ложное обнаружение preamble увеличивает занятость радиочипа, даже если полный пакет не принят.

## Sync word

Sync word помогает фильтровать кадры другой логической сети. Он не шифруется и не является паролем. Устройства с другим sync word могут всё равно создавать помеху, а анализатор способен определить параметры сигнала.

Если два независимых MeshCore-сегмента должны не видеть пакеты друг друга, нужно учитывать не только sync word, но и частоту/полосу. Разделение по PHY уменьшает логическое пересечение, но общий спектр остаётся общим.

## Explicit и implicit header

В **explicit header mode** PHY header содержит параметры, нужные для декодирования payload, в том числе длину. Это удобнее для пакетов переменного размера.

В **implicit header mode** длина и часть параметров заранее согласованы. Он уменьшает overhead, но требует строгого совпадения конфигурации.

Конкретная MeshCore-сборка и radio wrapper определяют режим. Пользователь не должен менять его независимо на одном узле.

## CRC

LoRa payload CRC обнаруживает случайные ошибки передачи. Это не криптографическая целостность:

- CRC не защищает от преднамеренной подделки;
- правильный CRC не подтверждает отправителя;
- повреждённый кадр отбрасывается до MeshCore;
- MeshCore cipher MAC отдельно проверяет зашифрованный payload.

Advert дополнительно содержит Ed25519 signature, потому что его публичные данные должны быть проверяемыми без общего секрета.

## Receive cycle

В постоянно слушающем узле радиочип после `begin()` переходит в RX. Драйвер периодически проверяет IRQ или получает callback.

Типовая обработка:

```text
RX raw bytes
→ журналирование RSSI/SNR
→ выделение Packet из pool
→ parse header/transport/path/payload
→ проверка версии и длины
→ расчёт score и estimated airtime
→ немедленная или отложенная обработка
→ release либо очередь ретрансляции
```

В текущем `Dispatcher.cpp` flood-пакет может попасть в delayed inbound queue: слабая по score копия ждёт, а сильная обрабатывается раньше. Если за время ожидания сеть уже увидела ту же передачу, поздняя копия будет подавлена как duplicate.

## Transmit cycle

Исходящий пакет сначала находится в outbound queue. Перед передачей диспетчер:

1. пополняет duty-cycle budget;
2. проверяет, наступило ли `next_tx_time`;
3. смотрит, не занят ли радиочип приёмом;
4. при CAD/busy откладывает попытку;
5. сериализует MeshCore packet;
6. оценивает максимальное время TX;
7. вызывает `startSendRaw`;
8. ждёт `TX_DONE` либо timeout;
9. учитывает фактический airtime;
10. возвращает radio в RX.

Пока идёт TX, узел не принимает другой пакет. Это одна из причин, по которой длинный airtime ухудшает не только пропускную способность, но и вероятность услышать входящее сообщение.

## CAD и состояние receiving

Channel Activity Detection пытается обнаружить LoRa-подобную активность. В реализации MeshCore решение о передаче связано с тем, сообщает ли radio wrapper состояние приёма/занятости.

Если канал остаётся занят, отправка откладывается на `getCADFailRetryDelay()`. После максимальной длительности busy-состояния может быть установлен error flag и выполнена принудительная попытка, чтобы зависший radio state не блокировал очередь навсегда.

CAD не гарантирует отсутствие коллизии:

- скрытый узел может быть не слышен отправителю;
- передача может начаться сразу после проверки;
- сигнал другого SF/BW может быть обнаружен не всегда;
- сильная не-LoRa помеха влияет иначе;
- два узла могут выбрать похожую случайную задержку.

## IRQ и ошибки

Типичные аппаратные события:

| Событие | Значение |
|---|---|
| `PREAMBLE_DETECTED` | найден кандидат LoRa-сигнала |
| `HEADER_VALID` | PHY header декодирован |
| `HEADER_ERROR` | заголовок повреждён или несовместим |
| `RX_DONE` | payload принят |
| `CRC_ERROR` | payload CRC не совпал |
| `TX_DONE` | передача физически завершена |
| `TIMEOUT` | операция не завершилась в пределах окна |

Точные имена IRQ различаются между SX127x, SX126x, LR1110 и библиотеками. MeshCore использует абстракцию radio wrapper.

## AGC и noise-floor calibration

Автоматическое управление усилением выбирает gain для текущего сигнала. Сильные внеполосные сигналы могут загнать front-end в неудачное состояние. В MeshCore предусмотрены hooks для noise-floor calibration и AGC reset.

Это не означает, что периодический reset всегда полезен. Частый reset может создавать пропуски RX. Настройка нужна только после воспроизводимого теста.

## Timeout передачи

Диспетчер оценивает airtime и задаёт expiry с запасом. Если `TX_DONE` не приходит, пакет освобождается и фиксируется ошибка. Такой timeout предотвращает вечную блокировку, но не гарантирует автоматическую повторную доставку на прикладном уровне.

Повтор пользовательского сообщения контролируется attempt/ACK-логикой, а не только radio timeout.

## Максимальный размер

MeshCore packet должен помещаться в `MAX_TRANS_UNIT`. При сериализации учитываются:

- 1 байт header;
- 4 байта transport codes при transport route;
- 1 байт path metadata;
- фактические path bytes;
- payload.

Большой path уменьшает пространство и увеличивает airtime. Даже если payload не превышает `MAX_PACKET_PAYLOAD`, итоговый пакет должен пройти общую проверку длины.

## Диагностика по слоям

| Симптом | Вероятный слой |
|---|---|
| Нет preamble/RSSI | частота, антенна, radio state |
| Header error | несовместимый PHY, слабый сигнал |
| CRC error | коллизия, помеха, низкий запас |
| MeshCore unsupported version | PHY принят, протокол несовместим |
| Invalid path length | повреждённый или несовместимый packet |
| MAC/decrypt fail | ключ, hash collision, повреждение payload |
| Нет ACK | путь туда или обратно, адресат, очередь, timeout |
